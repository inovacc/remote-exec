# Talos Linux Secure Communication Architecture — Implementation Brief for `remote-exec`

Research date: 2026-07-14. Scope: security + transport + identity design only (not Talos OS features).
Sources are cited inline; claims verified against talos.dev / docs.siderolabs.com and the siderolabs/talos repo.

`remote-exec` = a cross-OS controller/agent that runs build/deploy/analysis commands on remote
mac/linux/windows machines. The relevant question Talos answers well: *how does one identity-bearing
party securely drive privileged, potentially destructive operations on a remote machine, with a
trust model that is bootstrapped from a shared secret and enforced per-operation?*

---

## 0. TL;DR of the Talos model

- **One transport:** gRPC over **mutual TLS** on a single port (`:50000`). No SSH, no shell, no
  password auth, no bypass. The client cert *is* the identity.
- **One PKI root:** a per-cluster **machine CA** (`os` CA). Everything — node server certs and admin
  client certs — chains to it. `talosconfig` embeds `{CA cert, client cert, client key, endpoints}`.
- **One gateway:** **`apid`** is the only ingress. It terminates mTLS, authenticates + authorizes the
  caller, then fans out (proxies) to backends (`machined`, `trustd`, other nodes' `apid`).
- **Bootstrap by shared token:** worker nodes join by presenting a **join token** to **`trustd`**,
  which signs their *server* CSR against the machine CA. The token is the only secret that crosses
  the trust boundary; it is never used for client-cert issuance.
- **RBAC from the cert:** the X.509 **Subject `Organization` (`O=`) field encodes roles**
  (`os:admin`, `os:operator`, `os:reader`, `os:etcd:backup`). A gRPC interceptor maps `O=` → role set
  → per-method allow/deny. This is exactly the "can this instance do destructive ops?" gate.
- **Out-of-band overlay:** **SideroLink** = a point-to-point **WireGuard** management network; node
  identity = **node UUID + WireGuard public key**, admitted by a join token over a gRPC provision API.

## 1. mTLS + PKI model

### Mechanism
- **Machine CA (the `os` CA).** Generated by `talosctl gen config` (or `talosctl gen secrets`). A
  self-signed CA whose private key stays in the cluster secrets bundle / control-plane machine config.
  Everything Talos-API chains to it. (A separate Kubernetes CA exists but is out of scope here.)
- **Two leaf cert classes off that one CA:**
  - **Server certs** (node `apid` TLS identity): SAN = node DNS names + IPs. **Auto-managed and
    auto-rotated** by Talos; valid **~1 year**, renewed on reboot/upgrade or on schedule.
  - **Client certs** (`talosconfig` admin/user credential): identity in Subject, **roles in `O=`**.
    Valid **~1 year**; **user's responsibility** to renew via `talosctl config new`.
- **CA validity ~10 years**; leaf validity ~1 year (talos.dev cert-management).
- **Trust bootstrap:** `gen config` mints the CA + the first `os:admin` client cert offline, and bakes
  the CA into every node's machine config. Node ↔ node and client ↔ node both verify against the same
  embedded CA — there is *no* fallback, no password, no `--insecure` except pre-enrollment
  "maintenance mode".

### `talosconfig` (the client credential)
A YAML file containing, per context: `endpoints` (which nodes' `apid` to dial), `ca` (base64 machine
CA cert), `crt` + `key` (base64 client cert + private key). That single file is sufficient to
authenticate and is the unit of access sharing/revocation.

### Go libraries
- `crypto/tls` — `tls.Config{ Certificates, ClientCAs/RootCAs, ClientAuth: tls.RequireAndVerifyClientCert, MinVersion: tls.VersionInterface... }`
- `crypto/x509` — `x509.CertPool`, `CreateCertificate`, `CreateCertificateRequest`, cert parsing.
- Talos wraps these in **`github.com/siderolabs/crypto/x509`** (CA creation, CSR helpers,
  `PEMEncodedCertificateAndKey`, key types Ed25519/ECDSA/RSA). Reusable standalone.

### Adapt to `remote-exec`
- Ship a `remotexec gen` that creates one **agent CA** (Ed25519, cheap + small) + a first admin client
  cert. Embed the CA into every agent's config at enroll time.
- Agent presents a **server cert** (SAN = host/IP); controller presents a **client cert** whose `O=`
  carries its role. Both verify against the single embedded CA with
  `tls.RequireAndVerifyClientCert`. That is the entire trust model — copy it verbatim.
- Leaf validity days, not years, with rotation; CA validity years.

## 2. `trustd` — the certificate-issuance service

### Mechanism
- `trustd` is a small daemon (runs on control-plane nodes, **port `:50001`**) implementing the Talos
  **SecurityService** gRPC ("Root of Trust").
- **Node join / cert acquisition:** a joining node generates a keypair + a **server CSR** (subject,
  DNS names, IPs) and calls `trustd.Certificate(CSR)` over gRPC, attaching the **machine join token**
  as request metadata. `trustd` validates the token, signs the CSR with the **machine CA**, and
  returns the signed cert + CA bundle. The node now has a valid `apid` server cert.
- **Hard rule (verified in source):** `trustd` **only signs server CSRs, never client certificates**
  (commit `9eaf33f`, "never sign client certificate requests in trustd"). Client/admin certs are only
  minted by holders of the CA key or by `apid`'s authenticated `config new` path — a joining machine
  can never escalate itself to an admin identity.
- The **join token** is the single shared secret that authorizes enrollment; it lives in machine
  config (`machine.token`).

### Go libraries
- `google.golang.org/grpc` + `google.golang.org/grpc/credentials` (mTLS-or-token-guarded channel).
- `crypto/x509` + `siderolabs/crypto/x509` for CSR verification and CA signing.
- gRPC **metadata** (`google.golang.org/grpc/metadata`) to carry the token.

### Adapt to `remote-exec`
- Provide an **enrollment service** (can live inside the controller) exposing one RPC:
  `SignServerCert(csr, joinToken) -> cert`. Validate a **short-lived, single-use join token**, sign
  the agent's server CSR with the agent CA, return cert + CA. Never sign *client* certs here.
- Result: enrolling an agent needs only `{controller address, join token}` — no pre-shared cert.

## 3. `apid` — the mTLS gRPC gateway / proxy

### Mechanism
- **Single ingress on `:50000`**, gRPC over mTLS, client cert required. It is the *only* externally
  reachable Talos API surface.
- **Terminate + authenticate:** verifies the client cert chains to the machine CA (`ClientAuth:
  RequireAndVerifyClientCert`), then extracts identity/roles from the cert.
- **Authorize:** a gRPC interceptor enforces RBAC per method (see §7) before the call proceeds.
- **Fan-out / proxy:** `apid` forwards requests to local backends (`machined`, `trustd`) and — for
  multi-node ops — to *other nodes'* `apid`. `talosctl -e/--endpoints` picks which `apid` you dial;
  `-n/--nodes` picks which nodes ultimately answer. `apid` re-originates the proxied call over mTLS,
  so the fan-out is itself authenticated node-to-node. Talos uses a proxying gRPC handler
  (`grpc.UnknownServiceHandler` / transparent proxy) so backend service defs don't have to be
  re-declared in the gateway.

### Go libraries
- `google.golang.org/grpc`, `credentials` (mTLS), `grpc/metadata`.
- Interceptors via `grpc.ChainUnaryInterceptor` / `ChainStreamInterceptor` (Talos historically used
  `github.com/grpc-ecosystem/go-grpc-middleware`).
- Transparent proxy: a codec that passes raw `[]byte` + `UnknownServiceHandler` (pattern popularized
  by `mwitkow/grpc-proxy`).

### Adapt to `remote-exec`
- The **agent** *is* your `apid`: a single mTLS gRPC endpoint on the machine. Auth-interceptor →
  RBAC-interceptor → command service. You likely don't need node→node fan-out initially, but keep the
  "gateway terminates identity, then dispatches" shape so authz lives in exactly one place.
- If you later add controller-mediated broadcast to many agents, reuse the "gateway re-originates over
  mTLS" pattern rather than trusting a forwarded identity header.

## 4. `machined` / COSI — the API surface & privileged-op gating

### Mechanism
- **`machined`** is Talos's PID-1 / init and owns the privileged operations (reboot, shutdown, reset,
  upgrade, config apply, file ops). It exposes the **MachineService** gRPC behind `apid`.
- **COSI = Common Operating System Interface** (`github.com/cosi-project/runtime`) — a
  **resource + controller** runtime. System state (machine config, network status, certs, service
  status) is modeled as typed **resources** reconciled by controllers, and read via a COSI **State**
  gRPC API (`talosctl get <resource>`). It's the read/observe plane; MachineService is the act plane.
- **Gating of destructive ops:** every RPC is a named method on MachineService; the RBAC interceptor
  in `apid` (§7) checks the caller's role against that method. Destructive methods (`Reset`,
  `Reboot`, `Upgrade`, `Shutdown`, `ApplyConfiguration`) require `os:admin`/`os:operator`; read
  methods require only `os:reader`. Because every mutation is a distinct typed method, authorization
  is per-operation rather than "shell access = everything".

### Go libraries
- `google.golang.org/grpc` + protobuf-generated service stubs.
- `github.com/cosi-project/runtime` (resource state/controllers) — optional for you.

### Adapt to `remote-exec`
- Define your command surface as **discrete typed RPCs**, not one generic `Exec(string)`. E.g.
  `RunBuild`, `RunAnalysis`, `Deploy`, `FetchArtifact`, plus explicitly-gated destructive ones
  (`Reset`, `Rollback`, `RunArbitrary`). Per-method typing is what makes per-method authz meaningful.
- You don't need full COSI; a lightweight read-only `GetStatus`/`GetInfo` RPC (agent version, host id,
  cert fingerprint, current role) covers the observe plane.

## 5. SideroLink / WireGuard — out-of-band secure overlay

### Mechanism
- **SideroLink** = a point-to-point **WireGuard** management overlay so a controller can reach nodes
  even without routable access to their real IPs (NAT, bare-metal, cloud edge). Uses **IPv6 ULA**
  addressing on the overlay.
- **Provisioning flow:** the node generates its own WireGuard keypair, then opens a **gRPC connection
  to the SideroLink provision API**, sending `{node UUID, WireGuard public key, join token}`. If the
  token is valid, the server returns `{server WireGuard public key, two overlay IPv6 addrs (node +
  server)}`. The node configures its WG interface with the peer and connects; the API server tracks
  the peer in a **`ServerBinding`** resource (`address`, `publicKey`) and keeps installation-wide
  state (`installation-id`, server private key) in a `siderolink` secret.
- **Node identity = its WireGuard public key** (cryptographic peer identity), bound to its **UUID**.
  `grpc_tunnel=true` can even tunnel WG over the gRPC connection when raw UDP is blocked.
- The Talos **maintenance-mode API listens only on the SideroLink overlay** — management surface is
  reachable exclusively over the authenticated overlay, not the public network.

### Go libraries
- **`golang.zx2c4.com/wireguard/wgctrl`** — configure kernel/userspace WireGuard peers from Go
  (`wgtypes.Key`, `wgctrl.Client.ConfigureDevice`). Cross-platform.
- `golang.zx2c4.com/wireguard` (wireguard-go) for a userspace device (useful on Windows/macOS where
  kernel WG may be absent).
- `github.com/siderolabs/siderolink` (the provision service + library) and `google.golang.org/grpc`.

### Adapt to `remote-exec`
- Optional but powerful for agents behind NAT/firewalls: give each agent a WireGuard keypair; on
  enroll, agent sends `{host UUID, WG pubkey, join token}` to the controller's provision endpoint,
  which allocates an overlay IPv6 and returns the controller's WG pubkey. All subsequent mTLS gRPC
  then rides the overlay — you get a stable address per agent plus a second cryptographic identity
  (the WG key) independent of the TLS cert. Use **wireguard-go userspace** for portable mac/linux/win.

## 6. gRPC + protobuf patterns

### Mechanism
- **Service defs:** each daemon = one protobuf `service` with typed request/response messages
  (MachineService, SecurityService, COSI State). Strong typing is what lets RBAC key off method names.
- **Streaming for long-running output:** long ops use **server-streaming** RPCs — e.g. Talos'
  `Logs`, `Dmesg`, `Copy`, `Read` stream chunks back. Command output that arrives over time maps
  directly to `stream BytesResponse`.
- **Interceptors for auth:** a **unary + stream interceptor chain** runs `authenticate` (pull identity
  from the verified peer cert via `peer.FromContext` → `TLSInfo.State.PeerCertificates`) then
  `authorize` (role vs method) before the handler. Cross-cutting, defined once.

### Go libraries
- `google.golang.org/grpc`, `google.golang.org/protobuf`, `grpc/peer`, `grpc/credentials`,
  `grpc/metadata`, `grpc/codes`+`status` for typed errors.

### Adapt to `remote-exec`
- Define `RemoteExecService` in `.proto`. Make `RunCommand`/`RunBuild` **server-streaming**
  (`stream ExecChunk { bytes stdout; bytes stderr; int32 exit_code; }`) so build/deploy logs stream
  live. Put auth + RBAC in a `ChainUnaryInterceptor`/`ChainStreamInterceptor`. Read the client cert
  from `peer.FromContext(ctx)` — never trust a self-reported identity field.

## 7. RBAC / role model — the destructive-op gate

### Mechanism
- **Roles live in the client cert's Subject `Organization` (`O=`) field** (verified: docs
  "certificate subject's organization field is used to encode user roles"). One cert can carry
  multiple `O=` values → multiple roles.
- **Built-in roles:**
  - `os:admin` — all methods (incl. destructive).
  - `os:operator` — operational actions (reboot, shutdown, etcd backup/alarm) but **no config changes**.
  - `os:reader` — safe/read-only methods (list files, but not read their contents).
  - `os:etcd:backup` — only the `EtcdSnapshot` method.
- **Enforcement:** the `apid` gRPC interceptor extracts `O=` from the verified peer cert → builds a
  role set → looks up the required role for the invoked full method name in a static method→role map →
  `codes.PermissionDenied` if not satisfied. Authorization is **per method**, evaluated on every call.
- **Issuing scoped creds:** `talosctl config new --roles=os:reader reader.yaml` asks the cluster to
  **mint a new client cert whose `O=` is exactly those roles** — least-privilege credentials are just
  a cert with narrower `O=`. RBAC is enabled by default (v0.11+).

### Go libraries
- `crypto/x509` (read `cert.Subject.Organization`), `grpc/peer`, `grpc/status`+`codes`.

### Adapt to `remote-exec` — this is the core "can this instance do destructive ops?" answer
- Encode role in the controller cert's `O=` (e.g. `rex:admin`, `rex:operator`, `rex:reader`).
- Ship a static `map[fullMethod]requiredRole`. Destructive RPCs (`Reset`, `Deploy --prod`,
  `RunArbitrary`) require `rex:admin`; build/analysis require `rex:operator`; status requires
  `rex:reader`. The authz interceptor is the single chokepoint — an agent with a reader cert
  *cannot* invoke a destructive method even if it crafts the request, because the check is on the
  cryptographically-verified `O=`, not on anything the caller sends.
- To grant a controller destructive rights you **re-issue its cert with a wider `O=`**; to revoke,
  let the short-lived cert expire (or add a CRL/serial denylist).

## 8. Node identity & discovery

### Mechanism
- **Primary machine identity = node UUID** (SMBIOS/hardware UUID), used by SideroLink and discovery to
  uniquely name a machine.
- **Cryptographic identities:**
  - **TLS layer:** the node's **server cert** (SAN + fingerprint) chaining to the machine CA — proves
    "this is a legitimate cluster node."
  - **Overlay layer:** the node's **WireGuard public key** — the peer identity on the SideroLink net,
    stored in `ServerBinding{address, publicKey}`.
- A node is thus addressable/identifiable by **UUID + cert fingerprint + WG pubkey**, all bound at
  enrollment. "Ask for the ID of the other instance" = read its UUID and/or the SHA-256 fingerprint
  of its presented cert (available from the verified TLS peer chain).

### Go libraries
- `crypto/x509` + `crypto/sha256` (compute cert fingerprint from `cert.Raw`).
- host UUID: platform SMBIOS read, or a generated-and-persisted `github.com/google/uuid` value.

### Adapt to `remote-exec`
- Give each agent a stable **agent ID**: prefer host UUID, else generate + persist a UUID at first
  boot. Expose `GetIdentity() -> {agentID, certFingerprint, wgPubKey, roles}`.
- The controller pins/records `{agentID → certFingerprint}` at enroll time; on every connect it can
  assert the presented cert fingerprint matches the recorded one for that agentID (identity
  continuity / anti-swap). This directly implements "ask for the ID of the other instance and verify
  it."

## 9. Recommended adaptation for `remote-exec` (concrete minimal design)

**Transport.** One agent process per machine exposing a **single gRPC endpoint over mTLS**
(`RequireAndVerifyClientCert`, TLS 1.3, one port, e.g. `:51900`). No shell, no second auth path.
Long-running commands are **server-streaming** RPCs (`stream ExecChunk{stdout,stderr,exit_code}`).
Portable via `crypto/tls` + `google.golang.org/grpc` on mac/linux/win.

**Identity / PKI.** One **agent CA** (Ed25519) minted offline by `remotexec gen`, embedded into every
agent + every controller config.
- Agent holds a **server cert** (SAN=host/IP), auto-rotated, short validity (e.g. 90 days).
- Controller holds a **client cert**; its **role is in `O=`** (`rex:admin` | `rex:operator` |
  `rex:reader`). Credential file mirrors `talosconfig`: `{caPEM, clientCertPEM, clientKeyPEM,
  endpoints[]}`.

**Enrollment.** Controller runs a tiny **enrollment RPC**: agent generates keypair + server CSR, calls
`SignServerCert(csr, joinToken)` with a **short-lived single-use join token** in gRPC metadata;
controller validates token, signs CSR with the agent CA, returns cert+CA. **Never sign client certs
here** (Talos' `trustd` rule). Optionally the same call allocates a **SideroLink-style WireGuard
overlay** (agent sends `{UUID, wgPubKey}`, gets an overlay IPv6 + server wgPubKey) so agents behind NAT
remain reachable — use **wireguard-go userspace** for cross-OS.

**Auth gating (the destructive-op control).** Two interceptors in one chain:
1. `authenticate` — pull the verified peer cert via `peer.FromContext`, extract `O=` roles + compute
   cert fingerprint.
2. `authorize` — static `map[fullMethod]requiredRole`; destructive RPCs (`Reset`, `Deploy`,
   `RunArbitrary`) require `rex:admin`, build/analysis require `rex:operator`, status requires
   `rex:reader`; else `codes.PermissionDenied`.
The gate keys on the cryptographic cert, so a low-privilege controller physically cannot invoke a
destructive method. Narrow a controller by **issuing a cert with a smaller `O=`**; revoke by expiry or
a serial denylist.

**Node identity / "ask the other instance for its ID."** Each agent has a stable **agentID** (host
UUID or persisted UUID) and exposes `GetIdentity() -> {agentID, certFingerprint, wgPubKey, roles,
version}`. Controller pins `{agentID → certFingerprint}` at enroll and asserts it on every reconnect
(identity continuity). Fingerprint = `sha256(cert.Raw)`.

**Minimal Go dependency set.** `crypto/tls`, `crypto/x509`, `crypto/ed25519`, `crypto/sha256`
(stdlib); `google.golang.org/grpc` + `google.golang.org/protobuf` + `grpc/peer`/`metadata`/`credentials`;
optionally `github.com/siderolabs/crypto/x509` (CA/CSR ergonomics) and
`golang.zx2c4.com/wireguard` + `.../wireguard/wgctrl` for the overlay; `github.com/google/uuid` for
agent IDs.

**What to deliberately drop from Talos.** No etcd/K8s CAs, no COSI runtime (a read-only `GetStatus`
RPC suffices), no multi-node `apid`→`apid` fan-out until you actually need controller-mediated
broadcast. Keep: single mTLS gRPC ingress, one-CA PKI, token-bootstrapped server-cert enrollment,
`O=`-encoded per-method RBAC, UUID+fingerprint+WG-key identity.
